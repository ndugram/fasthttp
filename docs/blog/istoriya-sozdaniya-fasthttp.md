# Кто я

Меня зовут **NEFOR**. Я веб-разработчик и уже около **2 лет работаю с Python-фреймворками**, в основном с **FastAPI** и **Django**.

Мне всегда было интересно не просто писать код, а **разбираться как работают инструменты под капотом**: как устроены фреймворки, как обрабатываются HTTP-запросы и как строится архитектура библиотек.

Именно из этого интереса со временем и появился мой собственный open-source проект.

# Как я создал fasthttp-client

В сентябре 2025-го я случайно наткнулся на FastAPI и просто офигел. Ну реально, вот этот код:

```python
from fastapi import FastAPI

app = FastAPI()

@app.get("/")
async def root():
    return {"message": "Hello!"}
```

И это работает. Без всякой ерунды, просто и понятно. Я прям залип — как так можно сделать всё через декоратор и всё само под капотом происходит.

А в декабре я уже начал интенсивно изучать как этот фреймворк работает изнутри, как он генерирует документацию, как обрабатывает запросы, как Pydantic интегрируется прямо в эндпоинты. Чем глубже копал, тем больше понимал — за этой простотой скрывается огромная архитектурная работа.

## Почему не фреймворк

Сначала подумал: а давай тоже что-то такое сделаю, веб-фреймворк например. Но потом прикинул — там столько всего надо знать: HTTP/1.1, HTTP/2, HTTP/3, ASGI/WSGI, event loop'ы всякие, роутинг, валидация... Это ж годы опыта нужно иметь. Я точно не потяну такой проект в одиночку.

Так что идея создать веб-фреймворк отпала сама собой.

## И тут случилось видео на YouTube

Вечером того же дня смотрю ютубчик, попалось видео — сравнивают скорость requests, aiohttp и httpx. И там показывали всякие бенчмарки, графики, цифры. И я такой: а ведь можно сделать свой HTTP-клиент! Не просто очередную обёртку над aiohttp, а что-то своё, с нормальным API.

Захотелось сделать библиотеку чтобы:
- Была простой как requests
- Поддерживала async как aiohttp
- Логи были красивые, а не через print()
- И чтобы pydantic сразу работал, не надо было ничего делать

## Начал писать ночью

Сел вечером, начал кодить. Сначала сделал обёртку над aiohttp — хотел проверить идею, работает ли вообще. Потом перешел на httpx, потому что он стабильнее и удобнее. Ну и дальше постепенно добавлял фичи:

- Красивый логгинг сделал
- Pydantic валидацию прикрутил
- Fluent API запилил
- Мидлвары добавил
- HTTP/2 подключил через hyper-h2

И оно работает. Честно говоря, не ожидал что так получится.

## Что умеет библиотека

Пока я писал код, постепенно добавлял фичу за фичей. Сначала хотел просто сделать удобную обёртку, но потом понял — можно сделать гораздо больше. Вот что получилось:

**Декларативный стиль** — никакой ерунды с созданием сессий и ручным управлением контекстом. Просто декоратор — и запрос готов:

```python
from fasthttp import FastHTTP

app = FastHTTP()

@app.get(url="https://api.github.com/users/ndugram")
async def get_user(resp):
    return resp.json()
```

**Асинхронность и параллельность** — всё работает на [httpx](https://www.python-httpx.org/) и asyncio. Если у тебя 10 запросов, они выполнятся параллельно, а не по очереди. Это реально экономит время, особенно когда работаешь с кучей API.

**Зависимости (Dependencies)** — можно модифицировать любой запрос перед отправкой. Добавить токен авторизации, хедеры, логирование — всё через Depends. Точно так же, как в FastAPI:

```python
from fasthttp import FastHTTP, Depends

app = FastHTTP()

async def add_auth(route, config):
    config.setdefault("headers", {})["Authorization"] = "Bearer token"
    return config

@app.get(url="https://api.example.com/data", dependencies=[Depends(add_auth)])
async def protected(resp):
    return resp.json()
```

**Middleware** — глобальная логика для всех запросов. Кэширование, логирование, ретраи — всё что угодно. Есть даже готовый [CacheMiddleware](https://fasthttp.readthedocs.io/ru/middleware.html#cache), который кэширует ответы.

**Pydantic валидация** — ответы автоматически валидируются через Pydantic модели. Никаких ручных проверок:

```python
from pydantic import BaseModel
from fasthttp import FastHTTP

app = FastHTTP()

class User(BaseModel):
    login: str
    id: int
    html_url: str

@app.get(url="https://api.github.com/users/ndugram", response_model=User)
async def get_user(resp):
    return resp.json()
```

**Безопасность из коробки** — это вообще отдельная тема. Библиотека включает:
- Защита от SSRF (блокировка localhost и private IP)
- Circuit breaker — если сервер падает, запросы не летят в пустоту
- Маскирование секретов в логах (токены, пароли — всё скрыто)
- Лимиты на размер ответов
- Защита от опасных редиректов
- Таймауты

Всё это включается одной строчкой `security=True` (включено по умолчанию).

**HTTP/2** — поддержка современного протокола через [hyper-h2](https://hyper-h2.readthedocs.io/). Включается так: `app = FastHTTP(http2=True)`.

**CLI** — можно запускать запросы прямо из терминала без написания Python кода. Удобно для тестирования API.

**Красивый логгинг** — вместо `print("GET", url)` получаешь красивые цветные логи с временем выполнения, статусом кода и прочим:

```
FastHTTP started
Sending 3 requests
→ GET https://api.github.com/users/ndugram | headers={...}
✔️ GET    https://api.github.com/users/ndugram  200  145.23ms
Done in 0.23s
```

Всё это я запилил потому что самому было реально нужно. На работе постоянно дёргал разные API, писал одно и то же раз за разом. И думал — почему нет нормального инструмента?
